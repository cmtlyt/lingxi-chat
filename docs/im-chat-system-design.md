# IM 聊天系统设计文档

## 1. 产品定位

`lingxi-chat` 的初期定位不是做一个完整的“微信/飞书平替”，而是探索一个 **AI 辅助 IM** 产品：人与人聊天仍然是主体，AI 作为可被唤起的辅助层，帮助用户更好地表达、理解信息和维系关系。

产品核心判断：

- 聊天关系仍然发生在人与人之间，AI 不替代用户社交。
- AI 的价值不是单独开一个聊天机器人页面，而是融入私聊、群聊等真实沟通场景。
- 初期优先验证体验，不急于设计完整的大规模 IM 架构。
- 长期方向是 AI 社交 IM，而不是纯 AI 聊天工具或通用协作平台。

一句话定位：

> 一个以私聊和群聊为基础、以 AI 助手和 Agent 增强沟通体验的即时通讯产品。

## 2. 初期产品目标

第一版先做可演示、可感知的完整体验，目标是让用户感受到“这个聊天产品因为 AI 变得不一样”。

初期重点：

- 支持基础私聊和群聊。
- 用户可以正常发送、接收、查看文本消息。
- AI 可以在聊天过程中被用户主动唤起。
- AI 能结合当前会话上下文提供回复建议。
- AI 能对当前会话或未读消息做聊天总结。
- 用户有一个专属 AI 助理入口，可以主动提问，也可以接收摘要、提醒等主动消息。
- 群聊中可以出现可 @ 的 AI Agent，以机器人形式公开回复。

初期不追求：

- 完整好友体系。
- 大规模群聊。
- 复杂已读回执。
- 端到端加密。
- 多端强一致同步。
- 完整后台管理体系。
- Kafka、分库分表、多区域容灾等复杂工程架构。

## 3. MVP 能力边界

MVP 采用“私聊 + 群聊 + 四个 AI 薄闭环”的方式验证方向。

### 3.1 IM 底座

首版 IM 底座只需要支撑核心聊天体验：

- 用户登录。
- 会话列表。
- 私聊。
- 群聊。
- 文本消息。
- 历史消息查看。
- 基础未读提示。

这部分不需要一开始就做成完整通用 IM 系统，只要能支撑 AI 辅助体验验证即可。

### 3.2 回复建议

当用户不知道如何回复时，可以唤起 AI。AI 读取当前会话上下文，生成若干条可选回复建议。

这个能力主要解决“不会回复、不知道怎么表达”的问题，是 AI 辅助 IM 的第一核心体验。

### 3.3 聊天总结

当用户面对大量未读消息或长群聊时，可以让 AI 总结当前会话内容。

总结结果应帮助用户快速知道：

- 大家主要聊了什么。
- 有没有需要自己关注的内容。
- 有没有问题、结论或待办。

### 3.4 悬浮助理

用户可以在私聊、群聊等聊天窗口中唤起自己的 AI 助理。入口可以是悬浮按钮、小弹窗，长期也可以扩展为 Live2D 或语音形象。

悬浮助理偏即时辅助，适合用户在当前页面里快速提问、生成回复、解释上下文。

### 3.5 群聊 Agent

群聊中可以存在一个可 @ 的 AI Agent。它和悬浮助理不同：

- 群聊 Agent 的回复对群成员可见。
- 它更像群里的机器人或协作成员。
- 后续可以在后台配置不同 Agent 的能力、人格和使用范围。

MVP 阶段只需要支持最小闭环：群成员 @AI 后，AI 能结合当前群聊上下文给出公开回复。

## 4. 专属 AI 助理窗口

除了聊天窗口中的悬浮入口，长期还需要一个专门和 AI 助理对话的窗口。

这个窗口不是普通聊天页，而是用户的个人社交中枢：

- 用户可以直接向助理提问。
- 助理可以主动发送每日摘要。
- 助理可以提示哪些消息需要回复。
- 助理可以总结群聊关系和近期互动。
- 助理可以反馈定时任务、提醒或 Agent 执行状态。

这个入口让 AI 不只是在某个聊天页面里被动响应，而是逐渐成为用户长期的私人沟通助理。

## 5. 长期产品分层

长期演进可以分成三层：

### 5.1 IM 底座

提供稳定的私聊、群聊、消息、会话、未读、通知等基础能力。

这一层负责“人和人能正常聊天”。

### 5.2 AI 助手

面向单个用户，提供私有的、上下文相关的辅助能力：

- 回复建议。
- 聊天总结。
- 关系记忆。
- 情绪陪伴。
- 主动提醒。
- 专属助理窗口。

这一层负责“AI 帮用户更好地参与聊天”。

### 5.3 Agent 平台

面向群聊和场景，提供可配置、可扩展的 AI Agent：

- 群聊机器人。
- 场景化 Agent。
- 后台配置能力。
- Agent 权限与可见范围。
- Agent 执行结果回流到聊天或助理窗口。

这一层负责“AI 作为成员参与群体沟通”。

## 6. 阶段路线

### 第一阶段：体验原型

目标是验证 AI 辅助 IM 的核心感觉。

- 私聊和群聊可用。
- 文本消息可发送和查看。
- 回复建议可用。
- 聊天总结可用。
- 悬浮助理可用。
- 群聊 Agent 可 @。
- 专属助理窗口有基础问答能力。

这一阶段不追求工程体系完整，优先保证体验闭环。

### 第二阶段：体验打磨

目标是让 AI 辅助真正融入聊天流程。

- 回复建议更贴合上下文和语气。
- 总结能区分重点、问题和待办。
- 助理窗口支持每日摘要和主动提醒。
- 群聊 Agent 支持基础配置。
- AI 能理解当前用户、当前会话和近期消息。

### 第三阶段：关系与记忆

目标是形成 AI 社交 IM 的差异化。

- AI 记住用户偏好。
- AI 理解不同联系人和群聊的关系状态。
- AI 能辅助用户维系关系。
- AI 能根据不同场景调整回复风格。
- 陪伴感从工具能力逐步变成产品体验。

### 第四阶段：Agent 扩展

目标是让 AI Agent 成为群聊和社交场景中的可扩展能力。

- 支持多个 Agent。
- 支持后台配置 Agent 能力。
- 支持不同群聊绑定不同 Agent。
- 支持 Agent 执行任务并反馈状态。
- 支持更多场景，如社区、兴趣群、活动群。

## 7. 技术设计原则

当前阶段技术设计服务于产品验证，不提前进入大规模 IM 架构设计。

原则：

- 优先保证体验闭环，而不是一次性设计完整平台。
- IM 底座先简单可用，后续再补可靠性和扩展性。
- AI 能力先做调用闭环，再逐步增强上下文、记忆和权限。
- 私有助理和群聊 Agent 需要区分上下文边界，避免把私有信息暴露到群聊。
- 长期保留向完整 IM、AI 助手和 Agent 平台演进的空间。

## 8. 当前结论

`lingxi-chat` 当前更适合作为一个体验先行的 AI 辅助 IM 产品来推进。

第一版不要写成完整通用 IM 系统，也不要把重点放在复杂工程架构上。更合理的方向是：先用私聊、群聊、回复建议、聊天总结、悬浮助理、群聊 Agent 和专属助理窗口，验证 AI 是否真的能改善人与人之间的聊天体验。
